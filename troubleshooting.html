Codex-guided support

When something seems wrong, type troubleshooting in Codex

NoDrift troubleshooting happens inside the user's workspace, with the Codex agent checking the relevant files, rules, memory records, and boundaries before anything is changed.

The simple trigger

If something goes wrong while you are already working in your NoDrift workspace, send this to Codex:

troubleshooting

You can also say run troubleshooting. Codex should pause normal work, identify the problem type, inspect only what is needed, explain what it found, and ask for go before changing, moving, deleting, restoring, publishing, pushing, uploading, or editing anything.

Website role

This page tells users what to say and what the Codex agent should check. The website does not inspect local files and is not a live support bot.

Codex role

The Codex agent works inside the user's NoDrift workspace, where it can check project files, setup state, topic maps, corrective lessons, and local website or Git status when those checks are needed.

Quick triage

Common problems and what Codex checks

Start with the symptom. Codex should narrow the scope before touching files.

Setup

NoDrift may be in the wrong folder

Say troubleshooting: check whether NoDrift is installed correctly. Codex should check the active workspace, root instructions, governance folder, project memory, setup prompts, and whether the user is in the intended drive and project.

Startup

begin did not start correctly

Say troubleshooting: setup did not start correctly. Codex should check whether profile selection, topic-map choice, and optional verification happened in the expected order.

Governance

Codex is not following NoDrift rules

Say troubleshooting: Codex is not following NoDrift rules. Codex should review the latest instruction, active governance, approval status, source boundaries, and corrective lessons.

Verification

A claim sounds too complete

Say troubleshooting: verification or coverage problem. Codex should identify the claim, the implied scope, what was fully read or tested, and whether a coverage table is required.

Recovery

Work happened in the wrong chat or drive

Say troubleshooting: work happened in the wrong chat or troubleshooting: I think I am in the wrong folder. Codex should inventory source, target, duplicate, and uncertain locations before copying or moving anything.

Continuity

Topic map or compaction confusion

Say troubleshooting: topic map problem or troubleshooting: recover after compaction. Codex should compare the current summary against durable files and authorized memory records before relying on earlier decisions.

Privacy

Public and private files may be mixed

Say troubleshooting: public private boundary problem. Codex should stop before publishing, uploading, pushing, sharing, or sending anything and separate public files from private memory, governance, topic maps, investor notes, and incident records.

Publishing

GitHub or website state is unclear

Say troubleshooting: check before GitHub or publishing. Codex should separate local preview, local files, Git repository state, GitHub Pages, public metadata, visual QA, and approval status.

Stop points

What Codex must not do without go

Troubleshooting is allowed to inspect within scope. Consequential changes still need approval.

File safety

Do not delete or overwrite first

Duplicate-looking files, old folders, C drive copies, G drive copies, and GitHub versions must be inventoried before anything is deleted, replaced, or moved.

External actions

No publish, push, upload, or send by implication

Access is not approval. Local preview is not the public website. GitHub repository state is not the same thing as GitHub Pages publication.

Private records

Keep memory private

Living Topic Maps, branch maps, incident records, corrective lessons, source registers, and private working notes are not public website material unless the user explicitly authorizes external use.
