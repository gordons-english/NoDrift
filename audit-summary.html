What NODRIFT Checks

NODRIFT checks the points where AI-assisted work usually loses reliability. It is not continuous hidden monitoring. It is a governed review pattern that Codex applies when the work reaches a decision point.

  • Drift away from the user's latest instruction, approved scope, or active project.
  • Hallucinated facts, unsupported claims, stale assumptions, or invented source details.
  • Unauthorized actions such as editing, publishing, committing, uploading, sharing, deleting, or using outside accounts without approval.
  • Project contamination from old chats, unrelated folders, private material, or another project's facts.
  • Source authority problems when drafts, screenshots, old capsules, current files, and user instructions conflict.
  • Open decisions that are still being discussed and must not be treated as approved facts.
  • Memory continuity risk before long pauses, new chats, continuation capsules, handoffs, or compressed chat history.
  • Implementation changes that need a record of what changed, why it changed, and how it was checked.
  • External action boundaries before anything leaves the local workspace or affects a connected account.
  • Verification before public claims, file-completion claims, or audit claims are treated as ready.

How Governance Is Accessed

NODRIFT does not run as hidden surveillance software. It works through visible governance files, addendums, memory records, source registers, approval rules, and checkpoints that a Codex agent is told to read before acting.

In a governed workspace, the project instructions point the agent to the active addendums first. The agent reads the Codex-specific addendum for file access, tools, permissions, verification, publishing, and stop/go behavior. It then reads the universal governance for drift control, source fidelity, decision labels, no unauthorized rewriting, and no unauthorized creation.

The rules are actively applied at decision points: when a task begins, before files are inspected, before a factual claim is made, before edits are made, before old context is reused, before external action, and after work is finished for verification.

  • A new agent should read the workspace instructions, then the governing addendums, then the project memory files.
  • The agent should identify the current task, controlling sources, allowed files, approval status, and verification needed.
  • If the task would edit, publish, commit, upload, send, delete, or use an outside account, the agent must stop for explicit approval.
  • If a claim cannot be tied to a file, screenshot, browser state, tool output, or Don's instruction, it must be marked uncertain.

How This Differs From Agent Memory

Codex, Claude Code, and Gemini-style coding agents can use project instructions or memory files. That gives them context. NODRIFT adds an operating layer around that context: source authority, decision status, permission gates, risk tracking, evidence records, handoff rules, and verification before claims.

In practice, the agent still uses its native instruction system. Codex can use project instructions such as AGENTS.md. Claude Code and Gemini-style tools have their own project-context mechanisms. NODRIFT is designed to sit above those mechanisms as the project control system.

The result is not simply "more memory." It is governed memory: what is true, what is approved, what is uncertain, what is risky, what must be checked, what must wait, and what the next agent should read before acting.

Claim Boundary

These are audited local records, not a promise that AI can never make mistakes. Public claims should use the phrase "no confirmed drift" and "no confirmed hallucinations" for the audited period.
