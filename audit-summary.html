What NoDrift Checks

NoDrift checks the points where AI-assisted work usually loses reliability. It is not continuous hidden monitoring. It is a governed review pattern that Codex applies when the work reaches a decision point.

  • Drift means the AI response no longer follows the user's latest instruction, approved scope, or recorded project direction.
  • Hallucinated facts, unsupported claims, stale assumptions, or invented source details.
  • Unauthorized actions such as editing, publishing, committing, uploading, sharing, deleting, or using outside accounts without approval.
  • Project contamination from old chats, unrelated folders, private material, or another project's facts.
  • Source authority problems when drafts, screenshots, old capsules, current files, and user instructions conflict.
  • Open decisions that are still being discussed and must not be treated as approved facts.
  • Memory continuity risk before long pauses, new chats, continuation capsules, handoffs, or compressed chat history.
  • Topic contamination when a substantive tangent should be preserved in a separate Living Topic Map branch.
  • Implementation changes that need a record of what changed, why it changed, and how it was checked.
  • External action boundaries before anything leaves the local workspace or affects a connected account.
  • Verification before public claims, file-completion claims, broad every/all claims, or audit claims are treated as ready.
  • Coverage tables before full-review claims; targeted searches and syntax checks do not prove full coverage by themselves.
  • Candidate incidents that must stay visible before a final classification is supported.
  • Session review coverage before a clean-zero statement is allowed.
  • Repeated comments, previous-turn failures, and avoidably weak or incomplete responses that interrupt productive work.

How NoDrift Monitoring And Review Work

NoDrift uses explicit review points, not invisible automatic surveillance. Before substantive work, Codex checks the latest instruction, active scope, controlling sources, approval boundary, and verification required. After substantive work, it reviews whether the answer and actions stayed aligned and whether a suspected issue must be preserved.

A suspected problem enters the incident log as a candidate before anyone decides whether it is drift, hallucination, another incident, or not an incident. The session audit coverage register separately shows whether every session inside the evidence scope was actually reviewed.

This prevents an empty incident log from being presented as proof that nothing happened. Confirmed counts, candidate status, and review coverage are always reported separately.

The NoDrift review process has an affirmative duty to preserve and report an error it recognizes or reasonably suspects. The user does not have to discover every problem first.

The NoDrift Corrective Training Loop

When the NoDrift review process recognizes or reasonably suspects an error, it preserves the exact relevant user message and the workspace response or action. It then identifies the supported error classification, assigns confusion, time-loss, information-loss, and overall impact ratings, prepares the correct response, links the applicable governance rules, and identifies affected work.

The complete private incident record remains the evidence. The active corrective lesson tells future NoDrift-governed sessions what to do differently, and the review ledger records whether the correction was verified and whether the failure happened again.

Impact And Criticality

NoDrift applies a 1 to 5 impact rubric to each recorded incident. The user's only required incident event is Critical or Not Critical.

Critical stops the affected work and requires direction. Not Critical is recorded and corrected while already authorized work may continue.

The User's Work Stays Small

The user questions anything that does not sound right, makes important decisions, approves consequential actions, and answers Critical or Not Critical.

NoDrift performs the routine classification, scoring, governance linking, correction preparation, record maintenance, recurrence review, and reporting. The user can correct any NoDrift classification or score.

How NoDrift Reports Protect Continuity

Immediate alerts, task-completion reports, project checkpoints, periodic operations reports, and on-demand reports keep decisions, incidents, corrections, risks, and next actions visible without making the user maintain the system by hand.

The reports point back to complete private records. They do not replace the evidence or hide limitations through compressed summaries.

After chat compaction, a Post-Compaction Reconciliation Audit treats the compacted summary as provisional, compares it against durable project files and authorized memory records, labels what is verified or uncertain, and stops before forward work if important decisions or boundaries may be missing.

How NoDrift Governance Is Accessed

NoDrift does not run as hidden surveillance software. It works through visible governance files, addendums, memory records, source registers, approval rules, and checkpoints that the Codex agent uses before acting.

In a governed Codex workspace, the project instructions point the Codex agent to the active addendums first. The Codex agent reads the Codex-specific addendum for file access, tools, permissions, verification, publishing, and stop/go behavior. It then reads the universal governance for drift control, source fidelity, decision labels, no unauthorized rewriting, and no unauthorized creation.

The rules are applied at decision points: when a task begins, before files are inspected, before a factual claim is made, before edits are made, before old context is reused, before external action, and after work is finished for verification.

  • A new Codex agent should read the workspace instructions, governing addendums, project memory, authorized Living Topic Map files, error-training index, and active corrective lessons.
  • The Codex agent should identify the current task, controlling sources, allowed files, approval status, and verification needed.
  • If the task would edit, publish, commit, upload, send, delete, or use an outside account, the Codex agent must stop for explicit approval.
  • If a claim cannot be tied to a file, screenshot, browser state, tool output, or the user's instruction, it must be marked uncertain.

How NoDrift Living Topic Maps Protect Continuity

A parent Living Topic Map preserves the active project's path: objective, approved decisions, exact wording, open questions, boundaries, relevant files, risks, and next actions. When a substantive tangent appears, an authorized branch map preserves it without mixing it into the parent conversation.

Approved branch decisions that affect the parent are promoted back with their branch source and status intact. This guards against lost decisions, repeated explanations, and topic contamination while keeping the original chat available for details that were never recorded.

Living Topic Maps are private navigation records. They are not full transcript backups, controlling sources, Git branches, or permission to act.

The NoDrift error-training records protect a different kind of continuity: what went wrong, what the correct response should have been, what work was affected, and what future NoDrift-governed sessions must do differently.

How NoDrift Memory Management Differs

Modern LLM tools can remember parts of a conversation, follow project instructions, use files, and work inside coding or writing environments. That context is useful, but context is not the same thing as project control.

Memory can help an AI recall what was said. It does not automatically prove which source controls, which decision was approved, which claim was verified, which action needs permission, which tangent belongs in a separate branch, or which previous error must never be repeated.

The result is not simply "more memory." NoDrift memory management is governed memory: what is true, what is approved, what is uncertain, what is risky, what must be checked, what must wait, and what the next NoDrift-governed work session should read before acting.

The NoDrift Reception Layer

NoDrift does not work inside the LLM and does not change what the LLM transmits. The LLM drafts, reasons, writes, codes, or searches through its native platform. NoDrift works on reception: what the active work session and user accept, verify, record, correct, reject, or act on after output arrives.

NoDrift does not attach governance files, memory files, or corrective logs to every outgoing message or conversation turn. Its files stay in the workspace and guide reception, checking, recording, and reporting.

  • Source authority: which files, messages, screenshots, or user instructions may be trusted for each claim.
  • Decision status: what is approved, proposed, rejected, superseded, unresolved, or only background.
  • Approval boundaries: which actions require explicit permission before work continues.
  • Verification rules: what must be checked before something is treated as true, complete, fixed, or ready.
  • Continuity records: project state, checkpoints, handoffs, Living Topic Maps, and topic branches.
  • Error training: recognized mistakes become classified, scored, corrected, and preserved as future lessons.
  • Evidence reporting: clean-zero claims, candidate incidents, limits, and audit coverage stay separate.
  • Customization: communication style and token use can change without weakening governance.

How NoDrift Works Across LLM Tools

The same NoDrift method can be adapted to Codex, Claude Code, Claude, ChatGPT projects and agents, Gemini-style workspaces, DeepSeek-style workspaces, and other LLM tools when the work environment supports persistent instructions, project files, references, or repeatable workflow records.

In Codex, NoDrift uses workspace files, governance documents, memory templates, logs, and website or code artifacts to structure the receiving process: what to verify, what to record, what requires approval, what cannot be assumed, and what must be corrected when an error is found.

In Claude Code, the same logic would live in a Claude Code project structure: rules, source register, memory files, approval matrix, error log, and corrective lessons. NoDrift would not change Claude's model behavior. It would govern how the Claude Code session receives output, checks it, records decisions, and prevents unsupported claims from becoming project facts.

A later Claude adaptation would follow after Claude Code and would use the available Claude project, instruction, artifact, and memory surfaces to preserve the same reception-side discipline.

In ChatGPT projects and agents, a NoDrift adaptation can use project instructions, project-level governance files, canvases or topic maps, memory records, and training logs. ChatGPT may generate the answer, but NoDrift defines how the work session handles that answer after reception.

In Gemini-style and DeepSeek-style environments, the implementation changes by platform, but the receiving job stays the same: preserve the user's intent, separate verified facts from assumptions, keep decisions visible, and stop before actions that need approval.

Across all of them, the distinction is simple: the LLM transmits; NoDrift governs reception. NoDrift does not work inside Codex, Claude Code, Claude, ChatGPT, Gemini, or DeepSeek, and it does not attach NoDrift files to outgoing turns. The files stay in the workspace while the active session and user decide what is trusted, checked, remembered, and allowed to move forward.

What NoDrift Does That LLMs Usually Do Not

LLMs are powerful generators and reasoning partners. NoDrift adds the reception discipline that ordinary model output does not usually enforce by itself.

  • It separates a plausible answer from a verified project fact.
  • It separates a suggestion from an approved decision.
  • It separates a useful tangent from the active workstream.
  • It separates a private memory record from public release material.
  • It separates a clean confirmed count from incomplete audit coverage.
  • It separates a model's ability to act from permission to act.
  • It gives future LLM work sessions a required path back to the rules, sources, decisions, incidents, and corrective lessons.
  • It keeps governance files in the workspace instead of adding them to every outgoing turn.

Claim Boundary

The strongest evidence of reduced rework is usable work density: substantial work was created, organized, reviewed, corrected, and prepared without the project being consumed by drift correction, hallucination cleanup, repeated re-explanation, or unauthorized action.
