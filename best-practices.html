Codex operating habits

Best practices for working with Codex

Use these habits to keep long projects clear, reviewable, and governed from first idea to finished work.

Source Note

This page is organized around the NoDrift working protocol. Jason Liu's paper Codex-maxxing, dated 2026/05/10, is cited where it supports or extends a practice.

User habits

What the user does

The user separates planning from execution, uses voice when helpful, approves one task at a time, and keeps projects in clean workstreams. The user also questions anything that does not sound right and answers whether a reported incident is Critical or Not Critical.

Codex behavior

What NoDrift trains Codex to do

Codex should ask before acting, preserve approved wording, verify before reporting completion, and stop before external or destructive actions. When authorized, it should also maintain the Living Topic Map and create clean topic branches before a tangent contaminates the active track. It should report recognized errors, prepare corrections, maintain private corrective lessons, and provide concise operations reports. NoDrift does not add its files to outgoing user messages; it trains the work session to handle received output and records correctly.

Beginner

Start With Clear NoDrift Control Habits

These practices make Codex easier to guide before the work becomes complex.

User habit

Use voice for planning and review

Speak naturally when explaining a project, reviewing a page, or answering setup questions. Voice captures more context than short typed prompts.

Primary source: NoDrift working protocol. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

User habit

Read the AI's writing actively

Do not only wait for the answer. Read how Codex explains the work, structures sentences, names decisions, and uses precise vocabulary. This builds a practical feel for how AI systems organize information.

As you notice the pattern, speak back in the same clear structure: direct context, specific constraints, concrete examples, and explicit next steps. Users usually get better answers when their own communication becomes easier for Codex to parse.

Primary source: NoDrift working protocol.

User habit

Question anything that does not sound right

You do not need technical expertise or proof before asking for review. Question exaggeration, contradiction, subject changes, ignored context, unexpected claims, unnecessary repetition, and weak or incomplete answers critically and politely.

Optional user request: "That does not sound right. Review the response under NoDrift and report whether an error occurred."

Primary source: NoDrift working protocol.

User habit

Keep one project in one primary workstream

Avoid mixing unrelated projects in one thread. A clean workstream makes decisions, files, and next steps easier for Codex to preserve. When a useful tangent appears, keep it by moving it into a separate topic-map branch instead of developing it inside the main track.

Primary source: NoDrift working protocol. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

User habit

Use the side panel as a review surface

Review pages, documents, spreadsheets, PDFs, or previews in the side panel and give feedback on the same artifact Codex is changing.

Source: Jason Liu, Codex-maxxing, 2026/05/10.

Shared habit

Ask for an outline before implementation

During planning, say that Codex should not build yet. Ask for questions, understanding, and an outline before file work begins.

Primary source: NoDrift working protocol.

Shared habit

Use go for the approved task

Codex should end substantive turns with one clear next directive. When that directive needs permission, Codex should ask for go. The word go approves only the task just described.

Primary source: NoDrift working protocol.

Shared habit

Use /goal for directed completion

When you define a result and want Codex to keep driving toward it, start with /goal. Goal mode increases persistence, not permission; edits, publishing, uploads, commits, and external actions still need approval.

Primary source: NoDrift working protocol.

Shared habit

Require coverage before broad claims

When Codex says every page, all files, complete, ready, verified, public-safe, or publish-safe, require a coverage table. Targeted searches, diffs, syntax checks, and spot checks can support a review, but they do not prove full reading by themselves.

Primary source: NoDrift working protocol.

User habit

Type troubleshooting before fixing

When setup, folders, memory, governance, GitHub, or publishing state seems wrong, type troubleshooting in Codex before moving, deleting, overwriting, pushing, or publishing.

User habit

Separate discussion from file work

Use discussion chats for strategy, wording, and decisions. Use work chats for files, verification, and publishing steps.

Advanced

Preserve memory and decisions across long work

These practices keep complex projects from depending only on chat history.

Codex behavior

Write important memory to files

Store project status, decisions, open loops, risks, and preferences in editable files so useful context survives thread changes.

Primary source: NoDrift working protocol. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

Shared habit

Review memory as diffs

When Codex updates memory files, review what changed. This keeps memory concrete instead of letting vague impressions accumulate silently.

Primary source: NoDrift working protocol. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

User habit

Pin or preserve important workstreams

Keep important ongoing work easy to return to. Continuity is useful when a workstream has decisions, preferences, and unresolved loops.

Primary source: NoDrift working protocol. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

Shared habit

Give discussion chats controlled read access

A discussion chat may need to read or search a work chat, capsule, or memory file. Reading context is not permission to edit.

Primary source: NoDrift working protocol. Supporting reference: Jason Liu's shared-memory pattern in Codex-maxxing, 2026/05/10.

Codex behavior

Create full continuation capsules

Before moving a long project to a new thread, preserve project status, files, decisions, approved wording, risks, and next steps.

Primary source: NoDrift working protocol.

Codex behavior

Run post-compaction reconciliation

After chat compaction, compare the compacted summary against durable project files, authorized memory records, and the user's newest instruction before doing forward work that depends on earlier decisions.

Primary source: NoDrift context continuity gate.

Codex behavior

Maintain a Living Topic Map

After authorization, keep the active conversation navigable by recording important decisions, exact approved wording, open questions, boundaries, relevant files, risks, next actions, and branch relationships.

Primary source: NoDrift working protocol.

Codex behavior

Maintain the personal training and defense shield

Report every error you recognize or reasonably suspect. Preserve the relevant exchange, classify and score the error, prepare the correction, link the governing rule, identify affected work, maintain the private records, and ask the user only whether the incident is Critical or Not Critical.

Primary source: NoDrift working protocol.

NoDrift update

Keep corrective lessons current

When an error pattern is corrected, NoDrift can preserve the correction as a user-safe lesson. Add the current update pack to the private project memory area so future sessions can apply the newest correction rules without replacing the full governance package.

Primary source: NoDrift corrective-learning protocol. Download current update pack

Shared habit

Use Critical or Not Critical for escalation

The NoDrift review process performs the incident analysis and recordkeeping. The user decides only whether the affected work must stop for direction or whether the issue can be recorded and corrected inside the authorized task.

Primary source: NoDrift working protocol.

Codex behavior

Branch substantive tangents before they contaminate the main track

When a topic has its own objective, decisions, sources, or deliverable, create an authorized hierarchically named topic-map branch and register it in the parent. Topic-map branches are not Git branches.

Primary source: NoDrift working protocol.

Shared habit

Promote branch decisions with their source intact

When an approved branch decision affects the parent project, record it in the parent map with its branch source, approval status, and effect. Do not promote proposed or unrelated branch material.

Primary source: NoDrift working protocol.

Shared habit

Protect approved wording

When wording is approved, mark it clearly. Codex should not casually rewrite approved language during later edits.

Primary source: NoDrift working protocol.

Technical

Use the right surface for the work

These practices help experienced users choose tools, artifacts, and verification loops.

Codex behavior

Choose the right control surface

Use the in-app browser for local web review, Chrome for signed-in browser sessions, and computer control for GUI-only work.

Primary source: NoDrift working protocol. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

Shared habit

Prefer durable static artifacts when enough

A single static HTML file can be easier to inspect, share, and revisit than a heavier app when the task does not need a server.

Primary source: NoDrift working protocol. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

Shared habit

Use goals only with verification

Long-running goals need a clear finish line and a way to check success. Ambition without verification is not enough.

Source: Jason Liu, Codex-maxxing, 2026/05/10.

Codex behavior

Use monitoring loops when available

For reviews, comments, pull requests, inboxes, or feedback threads, scheduled checks can keep work moving while the user is away.

Source: Jason Liu, Codex-maxxing, 2026/05/10.

Codex behavior

Package repeated workflows

When a workflow is useful more than once, turn it into a reusable template, skill, checklist, or protocol so it does not have to be retaught.

Source: Jason Liu, Codex-maxxing, 2026/05/10.

Codex behavior

Pause before external actions

Publishing, committing, deleting, uploading, sending, buying, or sharing should require explicit approval unless that action was clearly included in the approved task.

Primary source: NoDrift working protocol.
